## Scope and Method

This document inventories the work required to make GitHub the source of truth for mooR again. It
covers repository data, GitHub Actions, release artifacts, container images, Debian and npm package
distribution, dependent repositories, local tooling, documentation, and the final Codeberg
retirement steps.

The repository scan excluded `.git/`, nested `.git/` directories, dependency caches,
`node_modules/`, `target/`, and local editor configuration confirmed to be untracked. It found 35
workspace files containing `Codeberg` or `Forgejo`, plus three additional workflows that are
Forgejo-specific by directory and runner configuration. Local Git metadata also identifies Meadow,
Meadow Flutter, and Cowbell as nested Codeberg checkouts. Host-side objects such as issues, pull
requests, releases, packages, secrets, webhooks, runner registrations, and branch protection rules
require a separate API export before cutover.

## Decisions Required

These decisions should be made before implementation begins.

1. **GitHub owner and repository layout**: The `timbran-project` organization now owns mooR.
   Consolidate Meadow, Meadow Flutter, and Cowbell into `timbran-project/moor`. Workflows should
   still derive the owner and repository from GitHub contexts rather than hard-code either value.
2. **Release boundary**: Publish 1.0.2 with the existing Forgejo release action before any monorepo
   or hosting changes. It becomes the final Codeberg-built release and rollback baseline.
3. **Debian repository host**: GitHub Packages does not provide an APT registry. The recommended
   target is `https://packages.timbran.org/debian`, backed by object storage and generated by
   `aptly` or `reprepro`. GitHub Releases should also carry every `.deb` as a direct-download path.
4. **npm registry**: The recommended target for the public `@moor/schema` and `@moor/web-sdk`
   packages is npmjs, provided the maintainers control the `@moor` scope. GitHub Packages is a poor
   default for these public client libraries because installs generally require authentication and
   the `@moor` scope may not match the selected GitHub owner.
5. **ARM64 release builders**: Use the standard `ubuntu-24.04-arm` GitHub-hosted runner unless the
   release rehearsal exposes a compatibility or performance problem. Retained self-hosted ARM64
   hardware and reproducible cross-builds on x86-64 remain fallback options.
6. **Collaboration history**: Decide whether to migrate all Codeberg issues and pull requests or
   only open items plus a static archive. GitHub's normal repository importer copies Git history,
   not issues or pull requests.
7. **Compatibility window**: Choose how long existing Codeberg container and Debian URLs remain
   readable. A 90-day read-only window is a reasonable default, but no new artifacts should be
   published there after cutover.

## GitHub Organization and Cost Model

GitHub Free for organizations is sufficient for the public consolidated mooR repository. Standard
GitHub-hosted Actions runners are free and unlimited for public repositories, including the standard
Linux ARM64 runners. Public GitHub Packages are free; GHCR container storage and bandwidth are
currently free under GitHub's published policy.

The migration does not require a paid plan. Costs can still arise from larger runners, excess
Actions artifact storage, private packages, Codespaces, and external services such as the
replacement APT repository. Configure organization budgets to stop usage at the free limit, use
short artifact retention, and do not enable larger runners without an explicit maintainer decision.

References:

- <https://docs.github.com/en/enterprise-cloud@latest/get-started/learning-about-github/githubs-plans>
- <https://docs.github.com/en/actions/reference/runners/github-hosted-runners>
- <https://docs.github.com/en/billing/concepts/product-billing/github-packages>
- <https://docs.github.com/en/repositories/creating-and-managing-repositories/transferring-a-repository>

Configure the organization before enabling migrated automation:

- Require two-factor authentication for members and register at least two organization owners with
  tested recovery methods.
- Configure zero-spend organization budgets and review billing contacts before the first migrated
  workflow runs.
- Set conservative base permissions, create maintainer and release teams, and restrict repository
  deletion and transfer to owners. Consolidation means repository-level write access covers every
  component.
- Set the organization Actions policy, default `GITHUB_TOKEN` permissions, fork pull-request policy,
  security settings, branch rules, environments, secrets, webhooks, and deploy keys.

The transfer from `rdaum/moor` to `timbran-project/moor` is complete. GitHub transfers repository
issues, pull requests, releases, stars, watchers, secrets, webhooks, deploy keys, and Git history,
and redirects old web and Git URLs. Do not recreate `rdaum/moor`, because doing so permanently
removes those redirects. Complete these post-transfer checks:

- Audit issue assignees. During a personal-to-organization transfer, assignees who are not
  organization members can be cleared.
- Audit GHCR and npm package associations; package linkage varies by registry after a transfer.
- Review organization policies, Actions approvals, environments, secrets, webhooks, deploy keys,
  collaborators, teams, and branch rules.
- Replace or republish `https://rdaum.github.io/moor/`; GitHub does not redirect repository Pages
  sites during a transfer.

The separate Meadow, Meadow Flutter, and Cowbell repositories become read-only archives after their
histories and collaboration data are imported.

### Existing GitHub Workflow

`.github/workflows/release-publish.yml` is a partial GHCR implementation. It authenticates with
`GITHUB_TOKEN`, uses the Docker actions, and creates provenance attestations. It cannot be adopted
unchanged:

- The `amd64` and `aarch64` matrix jobs generate identical tags and push them concurrently. The last
  writer can replace the other architecture.
- It does not assemble a multi-platform manifest.
- It does not preserve the deployed `-x86_64` and `-aarch64` compatibility tags.
- It does not build Debian packages or create a GitHub release.
- Its manual `ref` input accepts a branch, tag, or arbitrary commit without enforcing the release
  branch policy.

### CI Builder Image and Runners

`deploy/Dockerfile-forgejo-builder` creates the unqualified local image `moor-ci:1.95.0`. It bundles
Rust, Node, Java, Leiningen, Graphviz, `licensure`, and the Docker CLI. The Forgejo workflows assume
that image is preinstalled on runners named `linux-x86` and `linux-arm64`.

GitHub-hosted jobs cannot rely on that local image name. Choose one of these approaches:

- Publish a versioned CI image to GHCR and use it only for jobs that do not need to invoke a host
  Docker daemon.
- Install the smaller job-specific dependency sets on standard GitHub runners and cache Cargo, npm,
  and Leiningen data.
- Register existing machines as GitHub self-hosted runners and label them by OS and architecture.

The recommended split is standard runners plus setup/cache actions for routine CI, a focused Elle
container or setup job, and Buildx running directly on a runner for container publication.

Define one release package manifest and reuse it in local and CI entry points. Maintainers must
decide whether the supported set is all eight packages: `moor`, `moor-daemon`, `moor-telnet-host`,
`moor-web-host`, `moor-curl-worker`, `moor-mcp-host`, `moorc`, and `moor-emh`.

The target GitHub release workflow should:

1. Trigger from a release tag and support an audited manual rerun.
2. Verify that stable tags are reachable from `v1.0-release` and that package versions equal the tag
   before publishing anything.
3. Build and test every Debian package on x86-64 and ARM64.
4. Upload immutable build artifacts between jobs, then attach checksummed `.deb` files to one GitHub
   release.
5. Build architecture images by digest, push digests to GHCR, and create multi-platform manifests
   only after both builds succeed.
6. Publish `ghcr.io/timbran-project/moor:<version>` and, for stable releases, `:latest` as
   multi-platform tags. Workflows should form this path from GitHub context so an owner-name change
   does not require YAML edits. During the compatibility window also publish `<version>-x86_64`,
   `<version>-aarch64`, `latest-x86_64`, and `latest-aarch64`.
7. Generate checksums, SBOMs, and provenance for release assets and images.
8. Publish the APT repository only after the GitHub release and both container platforms validate.
9. Use a GitHub Environment named `release` with approval, branch/tag restrictions, and access to
   the APT signing credentials.

`scripts/bump-version.sh` updates all Cargo and npm package versions, updates `ChangeLog.md`, and
creates an annotated tag. It does not push the tag or invoke a release workflow. The maintainer
runbook must state the expected order explicitly: update versions, review changes, validate, create
the tag, push the commit and tag, approve the release environment, and verify published artifacts.

Existing GitHub metadata does not imply that its refs or Codeberg collaboration data are current.
Before cutover, export and compare all four source repositories:

- All branches, tags, annotated tag objects, notes, and default-branch settings.
- Git LFS objects, if the host-side audit finds any despite the lack of local configuration.
- Open and closed issues, comments, labels, milestones, assignees, attachments, and cross-links.
- Open, merged, and closed pull requests, review comments, approvals, source branches, and patch
  refs.
- Releases, release notes, draft/prerelease state, attachments, checksums, and publication dates.
- Wiki Git history.
- Repository description, website, topics, license detection, social preview, and Discussions.
- Branch protection or rulesets, required checks, merge policy, Actions policy, environments,
  secrets, variables, deploy keys, webhooks, collaborators, teams, and runner registrations.
- Container tags and manifests, Debian package versions and suites, npm versions and dist-tags, and
  download/install instructions.

GitHub Importer copies source and commit history but not issues or pull requests. See
<https://docs.github.com/en/migrations/importing-source-code/using-github-importer/about-github-importer>.
Use a purpose-built migration script or API client for Codeberg metadata. Imported comments should
state the original author, timestamp, and source URL because API-created objects cannot impersonate
their original authors. Maintain an issue and pull-request number mapping for redirected links.
Apply component labels during import so the consolidated issue tracker remains usable, and retain a
static export even when an item is not recreated on GitHub.

For open pull requests, prefer preserving the contributor branch and recreating a real GitHub pull
request when possible. Ask the author to take ownership after import. If that is not possible,
import the patch and discussion as an attributed migration item rather than silently losing it.

Do not use an unchecked mirror push against the established GitHub repository: a destructive mirror
push can delete GitHub-only refs. Compare refs first, take bundles/backups of both sides, and push
reviewed branch and tag refspecs.

## Target GitHub Actions Layout

A reasonable destination layout is:

- `.github/workflows/ci.yml`: build, unit/integration tests, doctests, and clippy.
- `.github/workflows/license.yml`: license header validation.
- `.github/workflows/elle.yml`: consistency tests and failure artifacts.
- `.github/workflows/npm-dev.yml`: local schema generation and SDK validation followed by ordered
  registry publication when required.
- `.github/workflows/clients.yml`: React Meadow and Meadow Flutter checks, including generated
  binding drift after schema changes.
- `.github/workflows/cowbell.yml`: Cowbell compilation and headless tests when the core, compiler,
  VM, or relevant schemas change.
- `.github/workflows/meadow-release.yml`: React/Tauri release jobs for `meadow-v*` tags.
- `.github/workflows/meadow-flutter-release.yml`: Flutter platform release jobs for
  `meadow-flutter-v*` tags.
- `.github/workflows/debian.yml`: reusable package build and manual artifact build.
- `.github/workflows/release.yml`: validated tag release, GHCR manifests, release assets, and APT
  publication.
- `.github/workflows/book.yml`: only if GitHub takes ownership of documentation publication.

Every workflow should set explicit permissions, timeouts, concurrency, shell error handling, and
artifact retention. Fork pull requests must not receive publishing credentials. Branch rules for
`main` and `v1.0-release` should require the selected CI checks and prevent release workflow files
from being bypassed without maintainer review.

## Migration Sequence

### Phase 0: Secure and Configure the Organization

- Audit the Forgejo workflow and migration credentials, then retire them after their Codeberg work
  is complete.
- Confirm `timbran-project/moor` as the final repository and the three component subdirectory paths
  before changing repository or package metadata.
- Require member two-factor authentication, add a second organization owner, configure teams and
  base permissions, and set zero-spend budgets.
- Audit the completed transfer: old-path redirects, issue assignees, collaborators, packages, Pages,
  secrets, webhooks, deploy keys, environments, Actions policy, and branch rules.
- Select the APT host, npm registry, metadata scope, and Codeberg compatibility window.
- Assign a maintainer for mooR, Meadow, Meadow Flutter, Cowbell, documentation, and package signing.

### Phase 1: Publish the Final Codeberg Release

- Finish and validate the intended 1.0.2 bug fixes without monorepo or hosting changes.
- Publish the reviewed 1.0.2 tag using the existing Forgejo release action and its verified,
  narrowly-scoped repository secret.
- Verify and inventory the release, containers, Debian repository, and npm state as specified in the
  final-release checklist above.
- Announce 1.0.2 as the last Codeberg-built release and retain it as the rollback baseline.

### Phase 2: Capture and Freeze All Source Repositories

- Announce the planned write freezes and cutover window for mooR, Meadow, Meadow Flutter, and
  Cowbell.
- Export all refs and collaboration objects from the four Codeberg repositories, plus relevant data
  from the existing `timbran-project/moor` and `rdaum/cowbell` GitHub repositories, to timestamped
  machine-readable files.
- Back up every source and destination ref before history rewriting or synchronization.
- Record counts and identifiers so the post-import audit can detect omissions.
- Inventory and copy immutable release, container, Debian, and npm artifacts.
- Freeze the three component repositories after a final delta export; do not accept commits in both
  the standalone and consolidated locations.

### Phase 3: Consolidate the Monorepo

- Prepare disposable, verified clones of Meadow, Meadow Flutter, and Cowbell from their frozen refs.
- Rewrite their histories into `clients/meadow/`, `clients/meadow_flutter/`, and `cores/cowbell/`,
  namespace their tags, and retain old-to-new commit and tag maps.
- Merge the rewritten development histories into mooR `main`. Merge Meadow's supported stable
  history and tree into `v1.0-release` without altering the published 1.0.2 tag.
- Remove nested Git metadata, fetch scripts, component Forgejo workflows, generated build state, and
  registry-mediated internal build dependencies.
- Add the root npm workspace/build graph and Flutter schema-generation checks, then validate clean
  builds without Codeberg package credentials.
- Run server, schema, SDK, both client, Cowbell compile, and Cowbell headless tests against the
  consolidated tree before publishing it.

### Phase 4: Prepare the Consolidated Organization Repository

- Retain `timbran-project/moor` as the destination and confirm its GitHub-hosted metadata and refs
  are backed up before synchronization.
- Synchronize reviewed Codeberg mooR branches and tags, then publish the reviewed consolidated
  histories without deleting GitHub-only refs.
- Import mooR and component issues, pull requests, releases, wiki history, labels, milestones, and
  attachments with attribution, component labels, and identifier maps.
- Configure repository metadata, collaborators, rulesets, Actions permissions, environments,
  secrets, variables, and runners.
- Audit GHCR package associations and replace or republish the nonredirecting `rdaum.github.io/moor`
  Pages site at `timbran-project.github.io/moor` or the canonical `timbran.org` endpoint.
- Remove the GitHub description and README notice that direct users to Codeberg only at final
  cutover.

### Phase 5: Port Unified Automation

- Add the GitHub CI, license, Elle, client, Cowbell, and generated-binding workflows and make the
  applicable jobs pass on both maintained branches.
- Preserve independent component releases with namespaced tag triggers and path-aware jobs.
- Replace the current GitHub container prototype with digest-based multi-platform publication.
- Validate Debian and container builds on `ubuntu-24.04-arm` and use a fallback runner strategy only
  if that standard runner fails the release rehearsal.
- Consolidate Debian package selection and build it through one reusable workflow.
- Port schema and SDK publication with deterministic ordering after local workspace validation.
- Dry-run release jobs without publishing, then publish a disposable prerelease tag.

### Phase 6: Move Distribution

- Copy supported image tags to GHCR and verify both architectures.
- Create the new signed APT repository, publish existing stable packages, and test clean installs
  and upgrades.
- Establish the npm target, publish and consume test versions, then move the `dev` dist-tags.
- Copy historical GitHub release assets and verify checksums.

### Phase 7: Update Consumers and Documentation

- Change Compose and Kubernetes image references.
- Remove Meadow and Cowbell fetch instructions and document their in-tree paths.
- Change npm registry and package metadata for external consumers; verify the monorepo builds from
  local schema and SDK packages before disabling Codeberg publication.
- Change APT key and repository instructions.
- Change repository, issue, release, clone, and source-browser links.
- Update Meadow, Meadow Flutter, and Cowbell documentation to use `timbran-project/moor` directory
  links and explain their independent version tags.
- Update MCP and editor tooling to GitHub without committed credentials.
- Build the book and run a link checker over Markdown and generated HTML.

### Phase 8: Cut Over

- Pause Codeberg writes, perform the final ref and metadata delta migration, and compare counts.
- Enable GitHub branch rules and required checks.
- Run a stable release or a release-candidate rehearsal entirely from GitHub.
- Make GitHub the documented source of truth and update `timbran.org` and Discord notices.
- Archive all four Codeberg repositories and the separate GitHub component repositories with
  prominent links to `timbran-project/moor` paths and artifact deprecation dates. Disable Forgejo
  workflows, webhooks, deploy keys, tokens, and runners.

### Phase 9: Retire Compatibility Endpoints

- Monitor Codeberg registry access and migration reports during the agreed window.
- Stop serving mutable Codeberg tags and package indexes first; retain immutable historical release
  downloads if policy permits.
- Remove `.forgejo/` and rename or remove `deploy/Dockerfile-forgejo-builder` after all GitHub jobs
  have passed on `main`, `v1.0-release`, and a release tag.
- Re-run the repository scan and permit remaining Codeberg references only in history, archived
  notices, or intentional compatibility documentation.

## Validation Matrix

The migration is complete only when all of these checks pass:

- The `timbran-project` organization requires two-factor authentication, has at least two owners,
  configured teams and base permissions, zero-spend budgets, and audited member and collaborator
  access.
- Old `rdaum/moor` web and Git URLs redirect to `timbran-project/moor`, and no repository or fork
  has been recreated at the old path.
- The Codeberg 1.0.2 release has the recorded source tag, artifacts, checksums, image digests,
  Debian packages, and successful install or upgrade checks.
- Fresh HTTPS and SSH clones of `timbran-project/moor` contain the expected mooR and namespaced
  component branches, tags, and tag annotations.
- `clients/meadow/`, `clients/meadow_flutter/`, and `cores/cowbell/` are ordinary tracked
  directories with no nested Git metadata, fetch step, or dependency on a sibling checkout.
- Component history is reachable through the consolidated repository, and old-to-new commit, tag,
  issue, and pull-request mappings have been retained.
- The separate Meadow, Meadow Flutter, and Cowbell repositories are read-only archives whose notices
  point to the correct `timbran-project/moor` directory.
- GitHub issue, pull-request, release, wiki, label, milestone, and attachment counts match the
  export or have documented exceptions.
- CI passes from a maintainer branch and a fork pull request without exposing secrets.
- Required checks and merge rules work on `main` and `v1.0-release`.
- The Elle job preserves useful artifacts on failure.
- `ghcr.io/timbran-project/moor:<version>` and `:latest` run on AMD64 and ARM64, and compatibility
  tags point to the expected platform manifests.
- All supported `.deb` packages exist for both architectures, install in clean systems, start their
  systemd units, and upgrade from the last Codeberg-published stable version.
- APT metadata signatures validate against the documented key fingerprint.
- A clean monorepo build generates TypeScript and Dart bindings once from the checked-in schemas,
  builds the SDK and both clients without Codeberg registry credentials, and detects generated-code
  drift.
- `@moor/schema` and `@moor/web-sdk` install from the new registry in a clean unaffiliated client
  environment with correct semantic dependency ranges.
- GitHub release assets have published checksums that match local and APT copies.
- mooR, Meadow, Meadow Flutter, and Cowbell CI run from the root repository, while component release
  tags publish only their own artifacts and do not write to Codeberg.
- Every shipped Compose and Kubernetes configuration pulls the new image and reaches its health
  checks.
- The mdBook build and link checker contain no active Codeberg dependency, and the selected
  `timbran-project.github.io/moor` or `timbran.org` canonical publication URL works without relying
  on `rdaum.github.io/moor` to redirect.
- A final tracked-tree and reachable-history scan finds no plaintext tokens and no unintended
  Codeberg or Forgejo operational references.

## Rollback

Keep Codeberg read-only and keep its immutable artifacts available until the validation matrix has
passed and the compatibility window ends. If GitHub publication fails during cutover:

1. Stop new GitHub releases rather than overwriting partial tags.
2. Restore the previously recorded GHCR manifest and APT repository snapshot.
3. Re-enable Codeberg writes only if policy permits and the maintainers explicitly choose to abort
   the cutover.
4. Reconcile any GitHub-only issues or commits before a second attempt.

Rollback must not restore credentials retired after migration or make Codeberg and GitHub
simultaneously writable sources of truth. Keep the organization repository location and old-path
redirect stable while repairing automation or distribution endpoints. The published 1.0.2 tag and
artifacts remain the runtime rollback baseline; do not attempt to restore the old multi-repository
development layout after new monorepo commits have been accepted.
